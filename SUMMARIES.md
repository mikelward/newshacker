# SUMMARIES.md â€” AI summary card layout

The thread page renders two AI summary cards above the comments list:
an article summary (`SummaryCard` in `src/components/Thread.tsx`) and
a comments summary (`CommentsSummaryCard`, same file). Both are
generated by Gemini and fetched asynchronously through React Query.

## Current approach: render nothing while loading

Each card returns `null` while its query is in flight and mounts the
real content the moment data arrives. No skeleton, no `min-height`
reservation, no `aria-busy`, no layout telemetry.

This is a deliberate simplification from an earlier skeleton-with-
reservation design (see *Alternatives considered* below). The bet is
that the shared server-side cache makes the "data arrives after the
thread paints" window small enough in the common case that the
one-frame pop-in is cheaper to tolerate than the machinery required
to avoid it:

- Summaries are shared-cached in Upstash Redis (see SPEC.md Â§ *Shared
  server-side cache*). Warmed summaries return in single-digit ms.
- `StoryList` warms both `/api/summary` and `/api/comments-summary` on
  row-in-view via `warmFeedSummaries` (`src/lib/feedSummaryWarm.ts`).
- `prefetchFeedStory` warms summaries for `score > 100` rows;
  `prefetchPinnedStory` and `prefetchFavoriteStory` warm at pin /
  favorite time.

By the time a reader taps a popular headline, the summary query
usually resolves before â€” or at most a frame after â€” the thread
commits. Cold misses (fresh story, first viewer, cache miss) get a
one-frame pop-in as the summary arrives; we accept the CLS cost.

## Alternatives considered

### Skeleton-first render with reserved height

The previous approach. Each card held a ref, measured its
`clientWidth` via ResizeObserver, canvas-measured the summary font's
average char width, estimated the wrapped line count for a fixed
character budget, and set a `min-height` inline style so the card
never shrank on arrival. A shimmering line skeleton filled the
reserved space until data landed, and `overflow-anchor: none` kept
scroll anchored to a following comment if the card's final height
still disagreed with the reservation.

- **Why removed.** Expensive to maintain: three tunable constants
  per card, a `summary_layout` Vercel custom event to retune them
  from live data, two helper modules (`src/hooks/useContentWidth.ts`,
  `src/lib/skeletonSize.ts`), and a separate test suite
  (`Thread.summaryLayout.test.tsx`). The pop-in it was preventing
  only happens on cache misses, which are rare once the server-side
  cache is warm.
- **When to reconsider.** If the cold-generation case dominates â€” a
  change in Gemini latency, a spike in fresh-story opens before the
  warmer catches up, or loss of the shared Redis cache â€” bring the
  machinery back. Git history has the full implementation.

### Block thread-page render until summaries are ready

Wait on the server (or in the client's loader) for both Gemini
responses before painting the thread page at all. No skeleton, no
reservation, no reflow â€” the page appears once with the real
summary already in place.

- **Why not.** Hard latency floor on the thread page equal to the
  slower of the two Gemini calls â€” p95 around a second for the
  article summary, several seconds for the comments summary on a
  cold miss. Unacceptable regression for the common case (a
  returning reader whose cache is warm), imposed to avoid a
  cache-miss reflow.
- **When to reconsider.** If Gemini latency drops by an order of
  magnitude (sub-100ms p95) and the service worker serves summaries
  inside the same request as the item fetch.

### Server-side prefetch of summaries

Run a background job that polls HN's top-stories feed and calls
Gemini on each new story at ingest, warming the cache before any
reader opens it.

- **Status: implemented.** The warm-summaries Vercel cron
  (`api/warm-summaries.ts`, documented in `CRON.md` and SPEC.md Â§
  *Scheduled warming and change analytics*) already does this for
  the top-30 feed slice. Combined with on-view warming from
  `StoryList`, most popular stories are summarized before the first
  reader taps in.

### Client-side summarization in the browser

Generate the summary on the client from the article HTML, via a
WebAssembly model or a browser-native summarizer API (e.g. Chrome's
`window.ai.summarizer`).

- **Why not now.** Current browser summarizers are either too heavy
  (WASM models blow the Lighthouse bundle target in SPEC.md Â§
  *Performance Targets*) or not yet broadly shipped (on-device APIs
  are experimental and Chromium-only). Moving generation to the
  client also loses the shared-cache benefit.
- **When to reconsider.** When a browser-native summarizer ships
  to stable across major browsers at zero download cost, it becomes
  a useful fallback for offline or Gemini-is-down cases.
